# Architekt — Internationalisation, RTL et baseline international

> Standards Architekt pour tout projet "global-ready".
> Référence : ADR-033, ADR-034.

## Doctrine

> **Si le projet a une langue alternative ou un marché secondaire potentiel → i18n by default.**
> Coût d'ajouter i18n après = 3-10× coût d'avoir prévu dès le début.

## Définitions obligatoires par projet

Chaque projet global-ready doit définir explicitement :

| Aspect | Exemple | Source |
|--------|---------|--------|
| Default language | `en` | Brief client |
| Supported locales | `en, fr, ar, zh, ja` | Brief client |
| RTL requirement | `yes` (si Arabe ou Hébreu) | Auto-déduit |
| Timezone model | UTC stocké, locale display | Standard |
| Currency model | USD/SGD/AED multi-devise ? | Brief client |
| Tax/VAT assumption | GST SG 9 %, VAT UE 20 %, etc. | Selon juridiction |
| Date/time formatting | ISO 8601 stocké, ICU display | Standard |
| Number formatting | Locale-aware (1,000.00 vs 1.000,00) | ICU |
| Address format | Per country (Singapore vs Saudi vs France) | Library |
| Phone format | E.164 stocké, libphonenumber display | Standard |
| Legal entity | Architekt SG → quelle entité opère ? | Selon contrat |
| Data hosting region | SG / EU / UAE / US | Cf. ADR-017 + COMPLIANCE.md |
| Support window | UTC+8 9-18h / UTC+0 / etc. | SOW |

**Livrable** : section "International Constraints" dans le brief projet (template `docs/INTAKE.md`).

## i18n / RTL baseline (requis)

### Frontend (Next.js / Astro / Nuxt)

- ✅ **Aucune string hardcodée** — toutes via fonction `t()` ou équivalent
- ✅ **ICU message format** (variables, plurals, gender, dates)
- ✅ Locale-aware dates (Intl.DateTimeFormat)
- ✅ Locale-aware currency (Intl.NumberFormat)
- ✅ **Layouts RTL-compatibles** (logical CSS : `padding-inline-start`, pas `padding-left`)
- ✅ Bidirectional text testing (mélange LTR/RTL dans paragraphe)
- ✅ Content expansion tolerance (allemand +30 %, arabe +25 %)
- ✅ **Fonts** : Latin + Arabe quand MENA ciblé (Noto Sans Arabic, IBM Plex Arabic)
- ✅ URL strategy : `/fr/about` ou `fr.example.com` (jamais query string)
- ✅ `lang` + `dir` attributes sur `<html>`
- ✅ Switcher locale visible

### Backend (FastAPI / NestJS)

- ✅ Messages traduits par clé (i18next, fluent, gettext)
- ✅ Emails templates par locale
- ✅ Notifications timezone-aware (envoi à 9h locale, pas UTC)
- ✅ Slugs URL : translit ASCII pour non-latin (Arabe, CJK)
- ✅ Recherche : indexation per-language (tokenization)

### Base de données

- ✅ Encoding **UTF-8** partout (BD, connexion, fichiers)
- ✅ **Collation** locale-aware si tri texte
- ✅ Champs texte multi-locale : table dédiée `translations(entity_id, locale, field, value)` ou JSONB
- ✅ Timestamps **UTC** stockés, affichage timezone client

| Stack | i18n library |
|-------|--------------|
| Next.js | `next-intl` ou `next-i18next` |
| Astro | `astro-i18n` ou natif Astro 4+ |
| Nuxt | `@nuxtjs/i18n` |

| NestJS | `nestjs-i18n` |
| WordPress | WPML ou Polylang |
| Shopify | Markets + Translate & Adapt |

## RTL (Right-To-Left) — checklist

Si projet inclut Arabe, Hébreu, Urdu, Persan :

- ✅ CSS **logical properties** (`margin-inline-start`, `inset-inline`)
- ✅ Tailwind v4 RTL support natif via `dir="rtl"`
- ✅ Icônes flippées si directionnelles (flèches, etc.) — `[dir="rtl"] .arrow-icon { transform: scaleX(-1); }`
- ✅ Number alignement (chiffres restent LTR dans texte RTL)
- ✅ Mélange LTR/RTL : tester paragraphes mixtes (URL en LTR dans texte AR)
- ✅ Forms : labels alignés selon direction
- ✅ Navigation : menus alignés selon direction
- ✅ Date pickers : adaptation calendrier (Hijri optionnel pour KSA)

## Multi-currency

| Cas | Approche |
|-----|----------|
| Affichage seul (display only) | Conversion temps réel via API (exchangerate.host gratuit) |
| Paiement multi-devise | Stripe multi-currency (Stripe convertit) ou Paddle |
| Comptabilité multi-devise | Stockage en devise base + devise transaction + taux |

**Important** : taxes (VAT/GST) souvent dépendantes du pays acheteur + vendeur — déléguer à Stripe Tax ou Paddle si possible.

## Multi-timezone

- **Stockage** : toujours UTC ISO 8601
- **Affichage** : timezone utilisateur (browser ou profil)
- **Notifications** : envoyer à heure locale utilisateur (ex : "9h du matin lundi")
- **Reports** : préciser timezone dans les exports
- **Booking** : afficher 2 timezones (utilisateur + service)

## Internationalisation avancée

### Adresses

Format varie par pays. Library : `address-format` ou Google Places API.

| Pays | Format typique |
|------|---------------|
| Singapore | Building, street, postal code (6 digits), Singapore |
| UAE | PO Box (souvent pas de street address), city, emirate |
| France | Numéro + rue, code postal + ville |
| USA | Street, City, State, ZIP, Country |
| Japon | Code postal, préfecture, ville, quartier, numéro (inverse Western) |

### Téléphones

- Stocker en **E.164** (`+6591234567`)
- Validation et formatage : `libphonenumber-js` (frontend) ou `phonenumbers` (Python)

### Noms

- Pas présumer "first name + last name" (cf. arabe, chinois, hongrois)
- Champs séparés `given_name`, `family_name`, `display_name`

## Test i18n / RTL

- ✅ **Pseudolocalization** : `Ẑẑ` (zalgo) ou Lorem ipsum Arabe pour tester layout
- ✅ Test screen reader VoiceOver/NVDA en arabe
- ✅ Visual regression Playwright avec snapshot per-locale
- ✅ Tests Lighthouse pour chaque locale critique

## Quand i18n n'est PAS justifié

- Pilote MVP single-market (mais préparer abstraction)
- Outil interne 1 entreprise 1 pays
- Site éphémère / campagne unique

Documenter le choix dans ADR projet.
